1.0 系统概述与核心架构
1.1 系统目标
本系统（EPJ）的唯一目标是，在多轮对话评测沙盒中，实时、动态、且科学地“计算”出共情对话的进度，并基于该计算决策对话何时应自然停止。
1.2 核心模型：共情向量模型 (EVM)
本系统基于一个三维“共情向量空间” (C-A-P Space) 运行：
- C 轴 (Cognitive): 认知共情（被理解）
- A 轴 (Affective): 情感共情（被共鸣）
- P 轴 (Motivational): 动机共情（被肯定/赋能）
1.3 核心架构：传感器-计算器-决策者 分离 (关键)
为确保评测的“信效度” (Validity & Reliability)，本系统严格遵循三层分离架构：
- Judger (判别模型 - LLM) | 传感器 (Sensor)
  - 职责：作为“质性分析师”。
  - 任务：禁止输出任何数字分数或决策。其唯一任务是读取剧本和对话，并根据“质性证据”填写由人类专家设计的《心理量表 (Rubrics)》。
  - 输出：已填写的“量表”（JSON 格式）。
- Orchestrator (编排器 - 代码) | 计算器 (Calculator)
  - 职责：作为“量化计算器”。这是一个纯代码组件。
  - 任务：禁止进行任何质性推理或最终决策。其唯一任务是读取 Judger 提交的“已填量表”，应用透明的《计分规则 (Scoring Keys)》来计算向量，并打包“状态数据包” (State Packet)。
  - 输出：“状态数据包”（JSON 格式）。
- Director (导演模型 - LLM) | 决策者 (Decision-Maker)
  - 职责：作为“智能决策核心”。
  - 任务：接收来自 Orchestrator 的“状态数据包”。分析这些量化数据，并做出最终的“进度决策”（停止/继续）和“行为指导”（给User Agent）。
  - 输出：决策指令（[STOP/CONTINUE]）和指导（Guidance）。

2.0 T=0 阶段：“起点”的科学量化
目的：在对话开始前，量化该剧本的“初始共情赤字” (P_0)。
2.1 Judger (传感器) 的任务
- 读取剧本，并完整填写附件中的**《A. 初始共情赤字量表 (IEDR)》**。
- 输出：Filled_IEDR.json（已填写的量表）。
2.2 Orchestrator (计算器) 的任务
- 接收 Filled_IEDR.json。
- 应用附件中的**《B. 初始赤字计分规则 (IED-SK)》**（一个透明的Python函数）。
- 计算并存储“共情轨迹”的起点：P_0 (初始赤字向量)。
  - 示例：P_0 = (C: -4, A: -8, P: -13)

---
3.0 T>0 阶段：“进展”的科学量化
目的：在每个K轮周期结束时，量化被测模型所提供的“共情增量” (v_t)。
3.1 Judger (传感器) 的任务
- 分析最新的K轮对话。
- 完整填写附件中的**《C. MDEP进展量表 (MDEP-PR)》**。
- 输出：Filled_MDEP_PR.json（已填写的进展量表）。
3.2 Orchestrator (计算器) 的任务
- 接收 Filled_MDEP_PR.json。
- 应用附件中的**《D. MDEP进展计分规则 (MDEP-SK)》**（一个透明的Python函数）。
- 计算出本周期的“增量向量” v_t。
  - 示例：v_t = (c: +1, a: +3, p: +1)

4.0 进度判断循环 (核心计算与决策)
这是系统的核心循环，严格分离了“计算”与“决策”。
4.1 Orchestrator (计算器) 的工作
- 时机：在 T>0 的每个K周期，收到 Judger 的Filled_MDEP_PR.json后。
- 执行的计算：
  1. 计算 v_t: v_t = calculate_v_t(Filled_MDEP_PR.json) (见 3.2)。
  2. 计算新位置 P_t: P_t = P_{t-1} + v_t。
  3. 计算“公差参数” ε: Epsilon = 1.0 (从配置加载，代表“高阈值”)。
# 这是全局的、由您定义的“科学参数”
EPSILON_CONFIG = {
    "high_threshold": 1.0,  # “高阈值”剧本，其“公差”严格，为1.0
    "medium_threshold": 1.5, # “中阈值”剧本，其“公差”标准，为1.5
    "low_threshold": 3.0   # “低阈值”剧本，其“公差”宽松，为3.0
}
  4. 计算“是否在区域内” is_in_zone:
  5. Python
is_in_zone = (abs(P_t.C) <= Epsilon) and \
             (abs(P_t.A) <= Epsilon) and \
             (abs(P_t.P) <= Epsilon)
  1. 计算“是否超时” is_timeout: is_timeout = (current_turn >= Max_Turns)。
- 打包并发送“状态数据包” (State Packet)：
- Orchestrator 不做决策。它将所有计算结果打包，发送给 Director (决策者)。
- JSON
{
  "current_turn": 5,
  "max_turns": 30,
  "P_0_start_deficit": "(-4, -8, -13)",
  "P_t_current_position": "(-3, -6, -10)",
  "v_t_last_increment": "(+1, +2, +3)",
  "is_in_zone": false,
  "is_timeout": false
}
4.2 Director (决策者) 的工作
- 时机：接收到 Orchestrator 发来的“状态数据包”。
- 执行的决策：
  - Director (一个LLM) 分析这个JSON数据包。
  - 它基于这些量化数据，做出智能的、非规则化的最终决策。
- Director (LLM) 的决策逻辑 (示例)：
  - IF is_in_zone == true -> THEN 决策 = STOP - SUCCESS。
    - （理由：计算器已确认轨迹抵达“解决区域”。）
  - IF is_timeout == true -> THEN 决策 = STOP - FAILURE。
    - （理由：计算器已确认超时。）
  - IF v_t 向量中有强烈的负值 (e.g., a: -5) -> THEN 决策 = CONTINUE (满足“测试”需求)，并输出 Guidance (给User Agent)：“在下一轮表达你的失望和被评判感。”
  - IF v_t 向量是强烈的正值 (e.g., p: +3) -> THEN 决策 = CONTINUE，并输出 Guidance：“在下一轮表达你的喜悦和被肯定的感觉。”
  - IF v_t 向量趋近于0 -> THEN 决策 = CONTINUE，并输出 Guidance：“在下一轮尝试结束对话或进行礼貌性收尾。”
- 最终输出：Director（LLM）输出最终的指令包：
  - 示例1：{"decision": "CONTINUE", "guidance": "向对方确认你被理解了。"}
  - 示例2：{"decision": "STOP - SUCCESS"}
